# Skills — рабочий процесс для umka.dev

Этот файл фиксирует, какие Claude Code скиллы использовать для задач в этом репозитории (UI-правки, новые секции, i18n, a11y/визуальные тесты) и в каком порядке. Цель — не тратить контекст на повторное решение "как проверять работу", а следовать одному и тому же пайплайну каждый раз.

## Шаг 0 (ОБЯЗАТЕЛЬНЫЙ): референс из styles.refero.design перед ЛЮБОЙ дизайн-работой

**Перед каждой работой с визуалом** (новая страница, редизайн, новая секция, смена стиля) — сначала референс из **https://styles.refero.design/** (проверено: доступен через WebFetch, 2026-07-18):

1. `WebFetch https://styles.refero.design/` — каталог 2000+ AI-readable дизайн-систем реальных продуктов (Linear, Notion, Duolingo, ElevenLabs, Mercury…), сортировка Trending/Popular/Newest. В промпте просить список стилей, подходящих под вертикаль задачи.
2. Страница конкретного стиля: `https://styles.refero.design/style/{UUID}` — отдаёт полный DESIGN.md: точные цветовые токены (hex + роль), типографику (шрифт/размеры/трекинг/line-height), радиусы, spacing, компоненты, do/don't-правила, surfaces. Пример: `/style/9130ad37-bf80-458f-b808-ac0ef6a8d1e9` (MindMarket).
3. Пользователь также может вставить готовый Style Reference (формат styles.refero.design) прямо в чат — так были сделаны Limón × FMR для Dastarkhan.
4. **Визуальная валидация референса через Playwright (ОБЯЗАТЕЛЬНО):** текстовый DESIGN.md не передаёт конкретные иллюстрации и композицию. Перед воспроизведением любого визуального элемента референса — открыть реальный сайт бренда Playwright-спеком, снять скриншот(ы), посмотреть их через Read() и только потом верстать. Текст описывает систему, скриншот показывает картинку — нужны оба.

**Правило применения — «фирменный стиль с вдохновением, не копия»:**
- Из референса берём: структуру палитры (роли цветов), типографическую логику (трекинг/масштаб/пары шрифтов), радиусы/плоскостность, фирменные приёмы (скобочные CTA, золотые линейки, ghost-обводки…), do/don't-дисциплину.
- Адаптируем под бренд проекта: свои хиксы под смысл (терракота/шаңырақ для казахского ресторана, а не лимон Limón буквально), свой контент, кириллические шрифты (`subsets: ["cyrillic"]` обязателен — Bodoni Moda, DM Sans, Outfit без кириллицы!).
- Каждый проект в итоге должен иметь СВОЙ узнаваемый стиль — refero даёт дисциплину системы, не готовую шкуру.
- do/don't-правила референса проверяем через axe: «золото — только штрих на светлом» из FMR оказалось буквально WCAG-требованием (2.47:1 текстом — фейл).

## Шаг 0.5 (ОБЯЗАТЕЛЬНЫЙ): humanized-текст в каждом проекте

Все пользовательские тексты (хиро, описания, подписи, FAQ, кейсы) проходят humanizer-проверку по канону humanizer-скиллов (blader/humanizer и производные, 29–53 паттерна). Русские аналоги главных ИИ-паттернов, которые ЗАПРЕЩЕНЫ:

- **«не X, а Y»** — контраст-формула («ставится человеком, а не выдаётся автоматически»). Заменять прямым утверждением с конкретикой.
- **Триады** — «Огонь, казан и тандыр.» Три однородных члена подряд — самый частый ИИ-ритм. Ломать: два, четыре, или перестроить.
- **Афоризмы-панчлайны** — «Каждое укрытие — временное.» Звучит как слоган из генератора.
- **Каскады тире** — больше одного тире на абзац почти всегда ИИ.
- **Идеальный параллелизм** — предложения-близнецы одинаковой длины и структуры.
- **Генерализации** — «каждый», «любой», «всегда» без фактуры.
- **Канцелярит** — «осуществляется», «является», «предоставляет».

Вместо них: конкретные детали (цифры, названия, «кто-то из команды смотрит вживую»), разговорные обороты, неровный ритм (короткое предложение после длинного), допустимая небрежность. Тексты пишутся так, как их написал бы владелец этого конкретного бизнеса, а не копирайтер-нейросеть.

## Дополнительный шаг: новый/переосмысленный demo-проект в карусели

Для карточек в `/demo/*` мало сменить акцентный цвет — так все проекты превращаются в один шаблон с перекрашенными карточками (это и было замечание пользователя про "иишный" вид Узла/Dastarkhan/FitPulse). Перед версткой нового или пере-дизайниваемого demo-проекта:

1. **`WebSearch`** — найти референсы конкретно под вертикаль и, если применимо, культурный контекст проекта (например: "kazakh national ornament web design", "craft marketplace textile knot layout", "kinetic typography fitness design 2026"), а не общие "web design trends".
2. Зафиксировать для проекта минимум 3 оси отличия от остальных карточек, не только цвет:
   - **Типографика** — отдельный шрифт (`next/font/google`, с проверкой `subsets: ["cyrillic", ...]`, т.к. весь контент на русском), а не общий Geist со всего сайта.
   - **Композиция/сетка** — не копировать структуру "hero → grid → footer" 1:1 между проектами (горизонтальный snap-scroll вместо grid, повёрнутые карточки вместо ровной сетки, ломаная/диагональная сетка вместо симметричной).
   - **Мотив/moushn, привязанный к смыслу проекта** — не абстрактная анимация "для красоты", а элемент, который отражает суть бренда (орнамент/шаңырақ для национального ресторана, нить-узел для "Узел", ECG-пульс + кинетическая типографика для фитнеса).
3. Реализовать, затем пройти обычный пайплайн ниже (`run` → `verify` → тесты → Read скриншотов).

## Пайплайн для любой правки UI/контента

1. **Правка кода** — как обычно (Edit/Write).
2. **`run`** — поднять dev-сервер и открыть/проскриншотить страницу в реальном браузере, а не только полагаться на код. Обязателен для любой задачи с визуальным эффектом (новая секция, overlay, карточка, тема, локаль).
3. **`verify`** — прогнать сквозной сценарий (не только `pnpm typecheck`/`pnpm build`, а реальный клик-путь: открыть карусель → открыть проект → переключить язык/тему и т.д.), убедиться, что фича действительно работает, а не просто компилируется.
4. **Playwright-сьюты проекта** (не отдельный skill, но часть того же пайплайна, зафиксированного в памяти):
   - `pnpm test:e2e --workers=3` (функциональные)
   - `pnpm test:a11y` (axe — критично, т.к. сайт позиционируется security/quality-first)
   - `pnpm test:visual` / `--update-snapshots=all` при изменении layout
5. **Read() скриншотов** — прежде чем закрывать задачу, реально посмотреть на сгенерированные/сделанные скриншоты глазами (см. `feedback_visual_verification_loop` в памяти), не доверять только зелёным тестам.
6. **`code-review`** (effort medium/high) — перед тем как считать нетривиальную правку законченной: ловит баги и лишние абстракции, которые тесты не поймают.
7. **`simplify`** — если правка разрослась (новые компоненты, дублирующийся код между секциями Services/Faq/Projects) — прогнать после code-review, отдельно от поиска багов.
8. **`security-review`** — обязателен для всего, что касается форм/контактов/ссылок/любого будущего backend-эндпоинта. Сайт продаёт security-first экспертизу автора (OSCP, пентест-опыт) — расхождение между заявленным позиционированием и реальным состоянием кода было бы особенно плохо здесь.

## Когда какой skill, коротко

| Skill | Когда | Почему именно тут |
|---|---|---|
| `WebSearch` | Перед версткой нового/переосмысленного demo-проекта в карусели | Без референса под конкретную вертикаль/культурный контекст проекты скатываются в один и тот же шаблон с разным акцентным цветом |
| `run` | Любая задача с видимым результатом (новая секция, overlay, анимация, тема) | Уже установленная практика сессии — код без визуального ревью не считается готовым |
| `verify` | Перед тем как сказать "готово" на нетривиальной фиче | Отделяет "компилируется" от "реально работает end-to-end" |
| `code-review` | После верификации, перед финальным ответом на некопеечные правки | Ловит баги/лишний код, которых axe и Playwright не видят |
| `simplify` | Когда правка добавила новые компоненты/паттерны (Services, Faq и т.п.) | Не даёт накапливаться дублирующимся секциям-карточкам-таблицам |
| `security-review` | Любая правка около форм, ссылок, будущих API/бэкенда | Сайт заявляет security-first позиционирование — код обязан соответствовать |
| `dataviz` | Только если появится график/диаграмма (сейчас не используется) | Нерелевантно для текущего контента, но держим в уме на будущее (например, визуализация тарифов/сравнения как диаграмма, а не только таблица) |

## Не относится к этому проекту (сейчас)

`artifact-design`, `claude-api`, `update-config`, `keybindings-help`, `fewer-permission-prompts`, `loop`, `schedule` — нет Artifacts, LLM-интеграций, кастомных hooks/keybindings или задач по расписанию в рамках этого сайта. Пересмотреть, если появится соответствующая фича (например, AI-виджет на сайте → `claude-api` становится релевантным).
